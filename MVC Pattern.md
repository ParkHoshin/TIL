```
(1) view (jsp)
- 모델을 표현하는 방법을 제공하는 사용자 인터페이스.
- 일반적으로 화면에 표시하기 위해 필요한 상태 및 데이터를 모델에서 직접 가져온다.

(2) Model
- 모든 데이터,상태 및 어플리케이션 로직이 들어있다.
- view와 controller에서 모델의 상태를 조직하거나 가져오기 위한 인터페이스를 제공하고 모델에서 자신의 상태변화에 대해서 옵저버들에게 알려주긴 하지만
  기본적으로 모델은 view 와 controller 에 별 관심이 없다.

(3) Controller
- view 와 controller 사이에서 위치하며 사용자로부터 입력을 받아서 그것이 모델에게 어떤 의미가 있는지 파악한다. 단순히 Model한테 전달하는 역할만
  하는것이 아니라, 사용자가 입력한 것을 해석해서 그것을 바탕으로 모델을 조작하는 임무를 맡고있다.
- view에서 controller의 역할을 직접 맡아도 되지만, 그렇게 하지 않는 것이 좋다!
  첫 번째로 view가 두 가지 역할을 맡기 때문에 코드가 복잡해지고 유지보수에 좋지 않다.
  두 번째로 view를 model에 너무 밀접하게 연관시켜야 한다는 문제가 있다.
  때문에 view를 다른 model하고 연결해서 재사용하기가 어려워진다.
- controller는 view 와 model의 결합을 끊어주는 역할을 한다.
  view 와 controller를 느슨하게 결합시켜 놓으면 더 유연하고 확장하기 좋은 디자인을 만들 수 있다.
  
* 사용자는 view(jsp)하고만 접촉할 수 있다.
- view는 model을 보여주는 창이라고 할 수 있다.
  재생 버튼을 누른다든가 하는 식으로 뷰에 대해서 어떠한 행동을 하면 view 에서 controller한테 사용자가 어떤 일을 했는지 알려준다.

* controller에서 model한테 상태를 변경하라는 요청을 한다.
- controller에서는 사용자의 행동을 받아서 해석한다.
  사용자가 버튼을 클릭한다든가 하는 어떤 행동을 하면 컨트롤러에서는 그것이 무엇을 의미하는지 분석하고 그 행동에 따라서 모델을 어떤식으로 조작해야
  하는지 결정한다.
  
* controller에서 view를 변경해달라고 요청할 수 있다.
- controller에서 view로 부터 어떤 행동을 받았을 때, 그 행동의 결과로 view를 바꿔달라고 할 수 있다.
  예를 들어, controller에서 인터페이스에 있는 어떤 버튼이나 메뉴를 활성화 또는 비활성화 시킬 수 있다.

* 상태가 변경되면 model 에서 view 한테 그 사실을 알린다.
- 사용자의 행동 때문이든 다른 내부적인 변화 때문이든,model에서 무언가가 변경되면 model에서 view한테 상태가 변경되었음을 알린다.

* view에서 model한테 상태를 요청한다.
- view에서 화면에 표시할 상태는 model로부터 직접 가져온다.
  예를 들어, model에서 view한테 새로운 곡이 재생되기 시작했다고 알려주면 view에서는 model한테 곡 제목을 요청하고, 그것을 받아서 화면에 표시한다.
  controller에서 view를 바꾸라는 요청을 했을 때도 view에서 model한테 상태를 알려달라고 요청할 수 있다.
  
  < 예제 >
  MVC 패턴을 공부하는 가장 좋은 방법은 MVC를 여러 개의 패턴이 함께 적용되어서 완성된 하나의 디자인으로 보는 것이다.
  model에서는 자신의 상태를 알려야 하기 때문에 옵저버 패턴을 사용해서 상태가 바뀔 때마다 view 와 controller에게 연락한다.
  controller는 view 의 행동에 해당하기 때문에 스트래티지 패턴을 이용해서 다른 행동을 원할 시 다른 controller로 교환하기만 하면 된다.
  view에서는 내부적으로 컴포지트 패턴을 써서 윈도우,버튼 같은 다양한 구성요소를 관리한다.
```
